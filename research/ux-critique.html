Metis Trail V2 — UX Critique & Design Recommendations

A thorough review of every visual, interaction, and structural element — with prioritized, implementable fixes

Executive Summary

The game has a solid structural foundation — the layout is clean, the color palette is grounded and period-appropriate, the typography (Rye + Alegreya) is well-chosen, and the single-file offline architecture works. But the interface is currently a skeleton. The map doesn't render without a network connection (Leaflet CDN), the narrative panel reads like debug output, the status bar is noise-dominant, overlays are bare-bones, mobile is a cramped afterthought, and there's zero atmospheric texture. The gap between "functional prototype" and "compelling game you'd play over coffee" is about 20 focused changes.

Rating the current state on narrative game UI principles:

  • Visual identity: 7/10 — Good palette and fonts, but no atmosphere
  • Information hierarchy: 4/10 — Everything competes for attention equally
  • Reading experience: 3/10 — Narrative panel is debug-text, not storytelling
  • Interaction design: 5/10 — Functional but undifferentiated buttons, no feedback
  • Mobile: 4/10 — Responsive but not designed-for-mobile
  • Accessibility: 3/10 — No ARIA, tiny tap targets, no focus management
  • Atmosphere: 2/10 — No sound, no animation, no spatial grounding

I. Critical Issues (Fix These First)

1. Leaflet is loaded from CDN — the map is always broken offline

The template loads Leaflet CSS and JS from unpkg.com. This is the single biggest UX failure. The entire game is supposed to be "offline-first" per the README, but the map component silently fails without a network connection. The map div renders as an empty #D8CCB0 rectangle.

Fix options:

A (recommended): Bundle Leaflet CSS into the inline style block. Use a minimal custom map renderer instead of Leaflet — a canvas-based terrain strip that draws the trail as a simple line with node dots. Total added weight: ~3KB. No external dependency. Draws instantly.

B: Include leaflet.css and leaflet.js in the dist/ folder and reference them locally. Adds ~150KB to the build output but preserves full mapping.

C (progressive enhancement): Build the custom canvas map as the baseline. If Leaflet loads (network available), swap to the interactive map. The game works either way.

2. The narrative panel is debug output, not storytelling

Current code in renderTravelLines():

function renderTravelLines(state, gameRef) { const here = gameRef?.getCurrentNode?.(); const next = gameRef?.getNextNode?.(); const lines = []; if (here) lines.push(`${here.name} (Day ${state.day})`); if (next) next.push(`Next: ${next.name}`); if (!lines.length) lines.push('On the trail...'); renderNarrative(lines); }

This produces: "Fort Garry (Day 1)" / "Next: St. Boniface" — raw data, not narrative. Compare with the source material we just compiled: "The prairie burned every afternoon... the oxen grew restless in the smoke." The narrative panel is the PRIMARY reading surface of the entire game. It needs to tell a story every single turn.

Fix: Every travel day should produce 2-3 lines of atmospheric text drawn from a pool of historically-grounded passages, matched to terrain type and season. Progression:

Phase 1 (immediate): Replace debug text with a pool of 10+ hand-written narrative lines per terrain type. Rotate with seed-based randomization. Each line should evoke the landscape, weather, or feeling of travel. No data readout in the narrative area.

Phase 2 (after events are populated): Narrative is the primary channel. Data (food, wear, crew) lives in the status bar. Narrative lives in the story. Never mix them.

3. The status bar is numeric noise — no visual hierarchy

Current status bar reads (all same visual weight):
DAY 1 / 6/15 / SUMMER / Fort Gary → St. Boniface / FOOD 30 / WEAR 0 / CREW Rested

Every piece of data has identical size and weight. The player's eye bounces across 8 items with no anchor. The critical numbers (food, wear) are buried between equally-prominent labels. On mobile at 390px, this bar wraps into chaos.

Fix — new status bar layout:

┌─────────────────────────────────────────────────────────┐ │ [CREST ICON] MÉTIS TRAIL Day 1 · June · 1878 │ │ │ │ ▓▓▓▓▓▓▓▓░░░░░░ Fort Garry → St. Boniface │ │ │ │ Food ████████░░ 25 Wear ██░░░░░░ 2 Crew: Rested │ └─────────────────────────────────────────────────────────┘

Key changes:
— Player identity (crest + title) anchors left
— Date is secondary information, smaller, right-aligned
— Trail progress bar (■■■■□□□□) replaces the "Node A → Node B" text readout. More spatial, less abstract.
— Food/Wear/Crew use compact bar or icon representations, not raw numbers in text
— Low food / high wear triggers color warning (amber → red) directly on the bar, not a separate class toggle

4. Button labels are flat and undifferentiated

Four identical buttons: Travel / Make Camp / Cart / Crew. They're the same size, same color, same weight. The primary action (Travel) gets no emphasis. Secondary actions (Cart, Crew as inventory views) look equally important. On mobile, they stack or compress.

Fix: Establish a visual hierarchy:

Primary action (Travel): full-width or 2x width, filled accent color, distinct style. This is what players press 80% of the time. It should feel like THE button.

Secondary actions (Camp): outlined style, smaller or standard width.

Tertiary actions (Cart, Crew): icon-only buttons with tooltip labels. These are reference views, not primary actions. Use a cart icon and a person icon. Saves horizontal space, especially on mobile.

5. Zero interaction feedback — the game feels dead

Clicking "Travel 1 Day" advances the game by one day. The status bar updates. The narrative lines change. But there is NO perceivable feedback: no button press animation, no screen transition, no sound, no number pop, no scroll effect. It feels like updating a spreadsheet, not traveling the Carlton Trail.

Fixes (lightweight, high-impact):
— Button press: CSS :active transform (scale 0.96) + color flash. Instant, zero JS.
— Day advancement: small fade-in on the new narrative text (CSS animation, 300ms)
— Dice roll on events: show the actual die result before revealing outcome. "Rolled 14 + 2 (mod) = 16 vs DC 12 — Success!" This is the single most important feedback moment in any narrative game.
— Travel: environmental audio cue (optional, Web Audio API — cart squeak, hoof beat, or wind). Even a 200ms sound clip transforms the feel.

II. Structural Improvements

6. The map paradigm is wrong for this game

The current interface devotes ~60% of vertical space to a geographic map (Leaflet with OpenStreetMap tiles). But the game is not spatial — players don't navigate freely. They move along a fixed linear path. A geographic map implies choice and exploration that doesn't exist. It adds complexity without gameplay value.

Fix: Replace the geographic map with a trail progress strip.

Fort Garry ●━━━━●━━━━●━━○━━○━━━━○━━━━○━━ ... SB SFX Port Ellice Edmonton

A horizontal progress bar with node markers. Current position highlighted. Visited nodes dimmed. Upcoming nodes faint. This communicates the same information (where am I, how far to go) in 40px of height instead of 60% of the screen. Frees up vertical space for the narrative panel (which is the actual game).

If you want to keep the map for flavor, make it a secondary element — a small inset or a toggleable overlay, not the primary visual.

7. Narrative area is too small — 28vh for everything below the map

With the map consuming most of the viewport, the bottom panel gets 28vh (roughly 250px on a 900px display). That's split between the narrative scroll area AND the control buttons. The actual readable text area is maybe 180px — 3-4 lines of text. This is not a reading experience. It's a ticker.

Fix: With the trail progress strip replacing the full map (saving ~40% vertical space), reallocate to:

  • Status bar: fixed 48px
  • Trail/Map strip: fixed 40px
  • Narrative panel: 45-50vh (up from ~18vh effective)
  • Controls: fixed 56px

On mobile, give the narrative a minimum of 200px readable height. The story is the game.

8. Event overlays are plain text on cream

The event overlay is a white card with a text block and buttons. No visual distinction between the event narrative and the choice text. No pacing. No sense of drama. Dice results aren't shown — you click a choice and the result appears in the narrative panel below, disconnected from the choice you made.

Fix — event overlay redesign:
— Event text area: italic, slightly larger font, warm background tint (not white)
— Choices: clearly separated as interactive cards with hover states
— Dice result: shown INSIDE the overlay after choice is made, with a "Continue" button. Don't dump the player back to the main screen mid-resolution.
— Source citation: small footnote at bottom of event card ("— John C. Schultz, 1894"). Builds trust, rewards curiosity, drives traffic to the research archive.

9. Settlement nodes have no mechanical identity

HBC post, Métis parish, mission, NWMP fort — these are all presented identically. Same overlay, same button layout, same visual treatment. A mission and a police fort should FEEL different.

Fix:
— Distinct icon per node type for the trail strip (cross for mission, beaver for HBC, fleur-de-lis for Métis, star for NWMP, etc.)
— Settlement overlay uses a different accent color per type (forest for HBC, cream/warm for Métis, blue-grey for NWMP, white for mission)
— Unique ambient text per node. Not just the description paragraph — a sensory detail. "The smell of wood smoke and bread" (Métis parish). "The HBC flag snaps in the wind" (HBC post). "A redcoat checks your papers, his breath visible" (NWMP).

10. What's working well

Before I bury you in criticism — the things that are genuinely good and should be preserved:

  • Color palette: #E8DCC8 cream, #1A1410 dark, #8B2500 red, #A67C29 brass. Grounded, warm, unmistakably period-Correct. This is the best part of the UI.
  • Typography pairing: Rye (headings) + Alegreya (body). Historically grounded, highly readable, distinctive without being gimmicky. Keep this.
  • Architecture: Single HTML file, vanilla ESM, no framework. Fast, portable, offline-capable. Don't add React or a build system.
  • Intro overlay: The journey setup paragraph is well-written. "It is June 15th, 1878. You stand at Fort Garry with a loaded Red River cart..." Good grounding. The tutorial text is clear.
  • CSS variables: Theming system is clean. Dark mode is already wired in (even if toggled off).
  • Source citation system: The schema already includes source objects with quote/author/work/year/url. This is rare in game development. Build onto it.

III. Mobile-Specific Issues

11. Mobile layout is responsive but not mobile-designed

The current mobile CSS reduces padding and font sizes reactively but doesn't fundamentally rethink the layout for a phone. At 390px width:

  • Status bar wraps into 2 rows (3-4 items per row, tiny text)
  • Four buttons in a single row compress to ~88px each — below Apple's 44pt minimum tap target
  • Map is still consuming 50%+ of vertical space
  • Narrative panel is ~30vh minus controls = maybe 150px of text
  • Overlay cards at 94% width with 20px padding = ~354px usable
Fix — mobile-first re-layout:

Option A (minimal): Stack controls 2x2 (2 rows of 2 buttons). Move status bar to a collapsible header. Use compact icon+number for stats.

Option B (recommended): Full mobile re-layout:

┌──────────────────────────┐ │ [≡] MÉTIS TRAIL Day 1 │ ← compact header │ Food ████░ Wear ██░░░ │ ← mini status bar ├──────────────────────────┤ │ │ │ NARRATIVE PANEL │ ← 55vh main content │ (scrollable text) │ │ │ ├──────────────────────────┤ │ [TRAVEL] [CAMP] │ ← primary actions full-width │ [🖼] Cart │ [👤 Crew] │ ← tertiary as icons ├──────────────────────────┤ │ ●━━●━━●━━○━○━━━━━○━━ │ ← trail progress strip └──────────────────────────┘

Mobile is 60%+ of casual web gaming traffic. Design for it first.

12. No touch-optimized tap targets

Choice buttons in event overlays are padding 8px 14px — fine for mouse, small for thumbs. Control buttons at 6px 16px padding are worse.

Fix: On touch devices (or at viewport < 768px), enforce minimum 44px tap targets on all interactive elements. Increase choice button padding to 14px 18px with 8px gap between them.

IV. Atmospheric & Sensory Enhancements

13. No weather/seasonal visual treatment

The game tracks season state but the UI never reflects it. Summer and early winter look identical.

Fix — seasonal atmosphere CSS (tiny codebase impact):
— Summer: Warm palette as-is. Occasional "heat haze" text shadow filter.
— Autumn:
Shift --clr-bg toward golden (#E8D4A0), add amber tint overlay. Crisper text.
— Early Winter: Cool palette shift (#D8D0C0), frost border treatment on the status bar (subtle white gradient at top). Condensation effect on text (very subtle text-shadow: 0 1px 2px rgba(255,255,255,0.3)).
— The transition between seasons should be a noticeable moment. Add a seasonal change event.

14. No cart squeal feedback

The squeal mechanic is tracked in state but never expressed. Players have no idea their cart is screaming across the prairie.

Fix:
— Visual: when squeal is high, add a CSS "shake" animation to the cart status indicator (1-2px vibration, subtle).
— Textural: narrative line appears: "'Squeak, squeak, squeak' — the cart's axle sounds like a gate in a gale."
— Audio (stretch): Web Audio API can generate a sine-wave squeal whose frequency/pitch scales with squeal value. 50 lines of code. Optional, muteable.

15. No ambient animation or life

Static UI = dead UI. Not motion for motion's sake, but environmental animation that signals "this is a living world."

Fix — minimal environmental animations:
— Wandering text: very subtle text-shadow drift on the title (simulates heat or wind)
— Map/trail animation: if keeping Leaflet, a gentle pulsing dot at current position. If using canvas trail, a subtle travel animation when moving between nodes.
— Seasonal particle: autumn = floating leaf emojis (🍂) drifting across the narrative panel. Winter = gentle snow (❄) using CSS @keyframes. Summer = nothing (stillness is also atmosphere). Implementation: 5 lines of CSS + a particle div.

16. Cart and crew overlays are bare-bones data views

The Cart overlay shows: "Pemmican Rations x30 (450 kg)". The Crew overlay shows: "State: Rested / Morale: 70 / Modifier: 1". This is a debug panel, not a game interface.

Fix — Cart overlay:
— Visual representation of the cart with items shown in their positions (left board, right board, cargo area)
— Weight meter showing current load vs 450kg max
— Color-coded items by type (food = green, repair = amber, trade = brass)
— Hover/tooltip on each item shows description and source citation
— Fix — Crew overlay:
— Named crew members (not abstract "State: Rested")
— Morale visualized as a facial expression icon or descriptive label ("Determined" / "Weary" / "Exhausted")
— History of recent events affecting morale (scrollable micro-log)

V. Accessibility & Production Quality

17. Zero ARIA / keyboard support

Every interactive element is a bare <button> inside an absolutely-positioned overlay. No aria-labels, no focus management, no keyboard navigation, no screen reader support. Overlay shown/hidden via display:none — focus can land on hidden elements.

Fix:
— Add aria-modal="true" to overlay containers
— Trap focus within active overlays (focus trap pattern)
— Add aria-live="polite" to the narrative panel (screen readers announce new text)
— All icon buttons need aria-label
— Ensure Escape closes all overlays
— Test with keyboard-only navigation (Tab + Enter)

18. Fonts loaded from Google CDN

Rye and Alegreya must be available offline. Currently not loaded in the template — the CSS specifies the font-family but doesn't load the web fonts. Browsers fall back to Georgia (which is fine for offline, but the Rye heading font never loads).

Fix options:
— Embed font data as base64 in the CSS (smallest fonts only, Rye is display font, ~40KB per weight)
— Self-host font files in dist/
— Accept Georgia fallback for offline and load Google Fonts when network available (progressive enhancement)

VI. End-Game & Scoring UX

19. End screen is underwhelming

"You Made It!" / "Journey Over" — one line of text, two stats (Days, Score), a Play Again button. After 20+ days of gameplay, the ending should feel like an arrival.

Fix — meaningful end screen:
— Longer narrative paragraph describing arrival at Edmonton. Describe the palisade walls rising from the riverbank, the sound of the bell, the weight of the completed journey.
— Show a journey summary: major events survived, events resolved, trades made, settlements visited — a "story so far" recap
— Display score breakdown: base + trade bonus + food bonus + crew bonus - days penalty - wear penalty
— Show the seed value ("Journey Seed: 2847 — share this to challenge a friend")
— Optional: "Historical Note" — a real passage about what the Carlton Trail meant to the people who traveled it.
