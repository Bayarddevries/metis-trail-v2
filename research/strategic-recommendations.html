Metis Trail V2 — Strategic Recommendations

Content systems, game design mechanics, project process, and growth strategy beyond the visual UX audit

Overview

This document covers what the UX critique and source compendium did not: the game design mechanics, the narrative systems, the build/deploy pipeline, the testing strategy, and the product decisions that will determine whether this project becomes a compelling experience or a well-structured prototype that nobody finishes. Every recommendation is grounded in what I found in the codebase, the node data, the source archive, and the three verified historical texts.

I. Narrative & Content Systems

1. The game needs a narrative engine layer between state changes and displayed text

Currently, renderTravelLines() in main.js pushes raw data strings. The event overlays display event.text (placeholder). There is no system that transforms what happened into what the player reads. Every good narrative game has this layer: a text generation system that takes game state and produces prose.

Build a simple text fragment assembler:

// Instead of: `${here.name} (Day ${state.day})` // Build: const fragments = []; fragments.push(pick(opening, { terrain, season, morale })); if (state.squeal > 30) fragments.push(pick(squeal_high, { squeal: state.squeal })); if (state.food <= 10) fragments.push(pick(food_low, { terrain })); if (state.wear >= 3) fragments.push(pick(wear_high, { node: here.name })); fragments.push(pick(season_detail, { season, terrain })); return fragments.join(' ');

Each terrain+season combination gets a pool of 15-20 text fragments. The seed-based RNG picks which ones appear. This alone transforms the game from a spreadsheet into a travel journal. The source compendium we built provides the raw material for every fragment pool.

2. Node descriptions are static — they should change based on game state

Every node in nodes.js has a single desc string. But a player arriving at Fort Ellice on Day 5 with full supplies should have a different experience than a player arriving on Day 30 with a broken cart and no food. The node description should be a function, not a string.

// Instead of: desc: "Midpoint resupply where the Assiniboine meets the Qu'Appelle..." // Build: desc: (state) => { const base = "Midpoint resupply where the Assiniboine meets the Qu'Appelle."; if (state.wear >= 3) return base + " The HBC factor eyes your battered cart with concern."; if (state.food <= 5) return base + " You eye the pemmican stores with desperate hunger."; return base + " Smoke rises from the fort's chimney. A welcome sight."; }

3. The "rumours" settlement action is underdeveloped — it should be a primary information channel

Currently "rumours" is just a settlement action that advances time by 1 day. But historically, the trail gossip network was how Métis communities shared intelligence about buffalo movements, river conditions, police activity, and trade prices. Rumours should give the player actionable intelligence that affects decisions:

  • "The buffalo have shifted west toward the Touchwood Hills" — hints at where food can be foraged
  • "The Assiniboine is running high — three carts lost at the Portage crossing" — warns of upcoming river event difficulty
  • "The Company is rationing pemmican at Carlton" — signals upcoming scarcity
  • "Riel's men are watching the trail south of Batoche" — NWMP/authority tension foreshadowing

Each rumour should be a structured data object with a mechanical effect (e.g., next river crossing DC +3, next forage roll +2, etc.) and a source citation from the research wiki.

4. Events need consequence chains — not just one-shot outcomes

Every event in the engine resolves immediately: roll, apply effects, done. But real trail events have consequences that ripple forward. A broken axle doesn't just add +1 wear — it means you're slower tomorrow, which means you consume more food, which means you might run out before the next settlement.

Implement a simple flags system (already in engine state as flags: {}):

// Event: "You ford the river but the cart bed soaks through." // Immediate: food -2 (wet rations) // Flag: { wet_rations: true } // Next 3 travel days: if wet_rations, food spoils at 2x rate // Resolution: at next settlement or camp, "Dry out the rations" action clears flag

This transforms events from isolated incidents into stories with arcs. The flags object is already in the engine — it just needs to be used.

5. The game needs a "journal" — a persistent log of what happened

Players should be able to review their journey. Not just the current state, but the story of how they got there. A scrollable journal overlay (accessible via a book icon) that records:

  • Each day's narrative fragment
  • Event outcomes with dice results
  • Settlement actions taken
  • Source citations for historical events

This serves three purposes: (a) players can re-read the story they're creating, (b) it provides context when they're trying to understand why their food is low, (c) it becomes a shareable artifact — "here's my journey" with seed value.

II. Game Design Mechanics

6. The economy is too simple — trade goods have no differentiation

Currently, "Bison Hide" and "Beaver Pelts" are both just "trade" type items with a count. But historically, these had very different economics:

  • Bison robes: 1.25 MB each, bulky, high volume. The staple of the independent freighter. Price stable but weight-heavy.
  • Beaver pelts: 3 MB each, compact, high value. The HBC's core commodity. Price fluctuated with European fashion.
  • Pemmican: 14¢/lb in 1878. Not a trade good — it's food. But it was also the currency of the trail. You paid for ferry crossings, guide services, and repairs in pemmican.

The trade system should have price variation by node. At Fort Garry, robes are cheap (buy low). At Edmonton, robes are expensive (sell high). At St. Paul, American goods are cheap. This creates the actual economic decision that defined Métis freighting: what to buy, where to sell, how to maximize the cart's 450kg capacity.

7. The crew system is too abstract — "rested/tired/exhausted" is not a person

The crew is currently a single state value. But the historical sources describe specific people with names, skills, and personalities. The Portage La Loche Brigade had named guides (Laurent Cadotte, Alexis L'Esperance, Jean Baptiste Bruce) with different wages and roles.

Implement named crew members:

  • Guide: +2 to navigation/forage rolls. Can be hired at settlements (costs 1 trade good or 5 food). Leaves after 3 nodes unless paid again.
  • Hunter: +2 to food foraging. Can hunt on plains terrain (requires ammunition).
  • Mechanic: +2 to repair rolls. Can do field repairs without tool kit.

Each crew member has a name drawn from historical sources, a portrait placeholder (initials in a circle), and a one-line bio. This makes the crew feel real and gives the "recruit" settlement action meaning.

8. The squeal mechanic needs a counterplay loop

Currently squeal just accumulates and makes noise. But the historical sources are clear about the counterplay: shaganappi grease. The mechanic should be:

  1. Squeal increases randomly during travel (terrain-dependent probability)
  2. High squeal attracts attention (encounter probability increases)
  3. Greasing the axle at camp or settlement resets squeal to 0 (costs 1 shaganappi)
  4. Without grease, squeal accelerates (dust + dry wood = worse each day)
  5. At squeal = 100, the axle is so loud it can be heard for miles — guaranteed encounter

This creates a resource tension: shaganappi is also used for repairs. Do you use it for grease (preventive) or save it for repairs (reactive)?

9. Food consumption is flat — it should vary by conditions

Currently: 1 food per day, always. But historically, food consumption varied:

  • Normal travel: 1 ration/day
  • Hard travel (mud, hills, forced march): 1.5 rations/day (round up)
  • Resting at camp: 0.5 rations/day (you're not burning as much, and you can forage)
  • Crew exhausted: they eat more (stress/desperation) = 1.5 rations/day

This makes the "Make Camp" decision meaningful beyond just morale recovery. It's also a food conservation strategy.

10. The season system needs mechanical teeth

Three seasons (summer, autumn, early winter) but only early winter has a mechanical effect (game over if not at final node). Summer and autumn should differ:

  • Summer (Jun-Aug): Normal travel. Rivers may be high from snowmelt. Forage is good. Prairie fire risk.
  • Autumn (Sep-Oct): Rivers lower (easier crossings). Forage declining. Days shorter (travel distance per day reduced by 10%). First frost events.
  • Early Winter (Nov): Game over if not at Edmonton. This is the real pressure — the calendar is the enemy.

The season transition should be a notable event. "The first frost silvered the grass this morning. Summer is over. You feel the urgency of the trail."

11. The scoring system doesn't reward the right things

Current scoring: base 1000 + trade_units*120 + food_bonus*12 + crew_bonus - days*8 - wear^2*40. This heavily penalizes wear (quadratic) and days (linear), but doesn't reward:

  • Historical events resolved: +50 per event (engagement with the story)
  • Source citations read: +10 per source clicked (engagement with the research)
  • Settlements visited: +25 per unique node (exploration)
  • Crew morale at arrival: morale * 5 (taking care of people)
  • Journal completeness: +10 per day logged (playing the full game)

Scoring should reward engagement with the historical content, not just speed and efficiency.

III. Build, Deploy & Testing

12. The build pipeline has a known CI failure that needs fixing

ISSUES.md documents: "CI log tar: dist: Cannot open: No such file or directory." The build script writes to dist/ but the GitHub Actions workflow apparently can't find it. This needs debugging before any deploy.

Check: does the GitHub Actions workflow run npm run build from the repo root? Does esbuild output to the right path? The build script uses process.cwd() which should be the repo root in CI, but worth verifying.

13. No test coverage for the engine

The TODO.md mentions "Add minimal unit tests for calendar and PRNG" but tests/ directory is empty. The engine has non-trivial logic (date advancement, PRNG, event resolution, scoring) that should be tested before adding content on top.

Priority tests:

  • Calendar: date advancement across month boundaries, leap year handling, season transitions
  • PRNG: seed reproducibility (same seed = same sequence), distribution sanity
  • Engine: food depletion → game over, wear max → game over, reaching final node → win
  • Scoring: verify formula produces expected values for known inputs

Vitest is already in package.json. Write 10 tests. Takes 2 hours. Prevents regressions forever.

14. The deploy target needs a decision

GitHub Pages is configured but not verified. The game should be playable at a URL. Options:

  • GitHub Pages (recommended): Free, automatic from dist/. Fix the CI, verify the URL, done.
  • Self-hosted: Any static host works. The single HTML file + JS bundle is trivially deployable.
  • Offline-first distribution: The game could be distributed as a single .html file (inline the JS bundle). Users download it, open in browser, play forever. No server needed. This is the most resilient option.

The offline-first distribution is philosophically aligned with the project. A Métis trail game that works without internet is a statement.

IV. Content Pipeline & Research Integration

15. The source wiki and the game need a tighter integration

The research wiki has 30+ entity pages, 20+ articles, and structured data. The game has 14 nodes and ~10 event slots. There's a massive content gap. Build a pipeline:

  1. Write a script that reads wiki markdown files and outputs game event JSON
  2. Each wiki article becomes 2-4 game events
  3. Each entity page becomes node description text + settlement flavor
  4. Source citations are auto-generated from wiki provenance

This means the research wiki becomes the content management system for the game. You write research → run the script → game has new content. No copy-pasting.

16. The game needs an "archive" view — a research mode

Players who finish the game (or who are curious before playing) should be able to browse the source material. An "Archive" button on the intro screen that shows:

  • All sources used in the game, with full citations
  • Direct links to archive.org for each primary source
  • Historical context paragraphs for each node
  • A timeline of the 1878 Carlton Trail journey

This positions the game as educational, not just entertaining. It also drives traffic to the research wiki and archive.org sources — which is good for the project's visibility and credibility.

17. Missing content: the return journey

The game is one-directional: Fort Garry to Fort Edmonton. But historically, the Carlton Trail was a round trip. The return journey (eastbound, spring 1878) had different challenges: spring floods, breakup ice, the opposite economic calculus (manufactured goods west, furs east).

This could be a "New Game+" mode. Same map, reversed direction, different events, different seasonal pressures. Doubles the content without doubling the node work.

V. Product & Growth Strategy

18. The game needs a shareable moment

Every good indie game has a moment players want to share. For Metis Trail, it should be the journey summary — a beautifully formatted "My Journey" card that shows:

  • The route taken with key events marked
  • Final score with breakdown
  • One standout moment ("You forded the Red Lake River in a horse-shoe pattern")
  • The seed value for challenge runs
  • A source citation for the most dramatic event that occurred

This card should be exportable as an image (html2canvas, 50 lines of code) or at minimum copyable as formatted text. Players share it → new players discover the game.

19. Seed-based challenge runs create replayability

The engine already supports seed-based play. Lean into this. After completing the game, show the seed and say: "Challenge a friend: can you beat seed 2847?" Leaderboard by seed. Weekly challenge seeds. This is how you get replayability without adding content.

20. The game should have a "historical mode" and a "story mode"

Historical mode: Strict. Historical prices, historical events only, no anachronisms. For the history nerds. Source citations always visible.

Story mode: Relaxed. More events, more dramatic, more forgiving. For the casual player. Source citations available but not foregrounded.

Same engine, different content pools and difficulty curves. Doubles the audience.

VI. Technical Debt & Code Quality

21. The schema.js typedefs are imported but never used for validation

nodes.js imports { Node } from schema.js but the schema file has no exports (it's all JSDoc typedefs). The import is dead code. More importantly, there's no runtime validation that events, nodes, or items conform to the schema. A malformed event object will silently produce broken UI.

Fix: either make the schema import meaningful (export validation functions) or remove the dead import. Add a validateEvent() function that checks required fields at load time. Run it in dev mode.

22. The debug UI and persistence modules are duplicated

debug.js and persistence.js both define saveGame, loadGame, and clearSave. The main.js imports from persistence.js, but debug.js has its own copies. This will cause bugs if they diverge. Consolidate into one module.

23. The engine creates a global window._metisGame — this is a debugging crutch that will cause problems

The game instance is exposed on window._metisGame and window.__METIS_DEBUG__. This is fine for development but should be stripped in production. The build script should have a --production flag that removes all window.__METIS* exports and the debug UI mount.

24. The PRNG is good but the seed input is never exposed to the player

The engine accepts a seed parameter but the UI has no way to input one. Add a "Journey Seed" field to the intro screen (collapsed by default, visible under "Advanced"). Players who want to share seeds or replay specific journeys can do so.

VII. Accessibility & Inclusivity

25. The game uses "half-breed" in source quotes — needs contextual framing

Several historical sources use the term "half-breed" (Fonseca, the barricade verses). This was the common term of the era but is offensive in modern usage. The game should:

  • Never use the term in game-authored text
  • When quoting historical sources that contain it, add a contextual note: "This term appears in the original 1869 source. The Métis Nation uses 'Métis' as its self-designation."
  • Default to "Métis" in all game text

This is both an accessibility issue and a historical accuracy issue — the Métis called themselves Métis, not "half-breed."

26. Color contrast needs verification

The current palette: #E8DCC8 on #1A1410 (status bar text on dark bg) — this is good contrast. But #8B2500 (accent red) on #E8DCC8 (cream) may fail WCAG AA for small text. Verify with a contrast checker. The brass #A67C29 on cream is likely borderline. If the game is educational and public-facing, WCAG AA compliance matters.

27. The game needs a text-only mode for screen readers

The map is purely visual. The narrative panel is text but not semantically structured (just divs with class "screen-text"). Add proper semantic HTML: <article> for narrative, <ol> for event choices, <dl> for status bar stats. The game should be playable with a screen reader even without the visual map.
