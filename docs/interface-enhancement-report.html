Improvement Options

These represent the biggest remaining clarity gains, ordered by impact-to-effort ratio.

Option 1 — Harden mobile touch targets

High impact · Low risk
  • Camp action buttons already 44px+ height
  • Secondary actions and close/dismiss targets should be audited and brought to ≥44px touch height
  • Overlay dismissal should have a clear, large tap target
  • No-visual-change rule: if a button already works, don't retouch it

Risk: Minor — only affects padding/height of smaller auxiliary controls.

Option 2 — Add concise status bar context

High impact · Medium risk

Status bar shows numbers, not states. Add tiny interpretive labels so a glance tells the story:

  • Food icon + numeric value + "low" label when ≤5
  • Wear icon with "rough" / "firm" / "broken" labels
  • Crew morale pill with rested / tired / exhausted state
  • Day counter with micro progress indicator

Keep labels to 1–2 words; occupy no more than 10% extra bar height.

Option 3 — Event result micro-summary

Medium impact · Low risk

After an event resolves, the result text often buries the actual delta. Add a one-line outcome summary:

+2 food · -1 wear · crew gained "tired" · 1 day passes

Players can verify their understanding in under 2 seconds. No extra clicks.

Option 4 — Location and route clarity

Medium impact · Medium risk
  • Show "Next: [name]" in larger type on travel overlay
  • Add distance indicator when 2+ nodes ahead become relevant
  • Make current node name persistent in the map header even after travel
  • Consider a minimal route breadcrumb: 2–3 prior nodes + current + next

Avoid map clutter — max 4 breadcrumb nodes, collapsed to "…" when longer.

Option 5 — Overlay action rubric consistency

Medium impact · Low risk

Right now, camp buttons show costs, event buttons show costs, but settlement/crafting buttons may not. Standardize the rubric:

  • Every clickable outcome shows cost/risk hints when relevant
  • Hover/focus reveals extra context on desktop
  • Same content for mobile tap and desktop hover

Option 6 — Typography and readability pass

Medium impact · Low risk
  • Set a single line-height rule for body text: 1.6–1.7 for mobile paragraphs
  • Raise overlay headline font size to at least 16px
  • Ensure overlay body text never drops below 13px
  • Use semi-bold for numeric values; regular for labels
  • Limit overlay text blocks to ~180 characters before truncating with "read more"

Option 7 — Encampment feedback and narrative variety

Low effort polish
  • Add 3–4 short scene blurbs that rotate when resting ("The fire crackles…")
  • Show dice roll result visually ("You rolled 17 vs DC 15 — refreshing rest!")
  • Track "last camp action" so replaying Rest shows a different flavor line

Data-light — no schema changes needed. Adds texture with minimal code.

Option 8 — Accessibility and contrast lift

Medium impact · Low risk
  • Verify overlay text against background passes WCAG AA on mobile
  • Add focus-visible outlines for keyboard navigation
  • Ensure color is never the sole indicator of state (add icon/text for food/wear warnings)
  • Test with system font zoom at 150%

Priority Recommendation

Recommended sequence:

  1. Mobile touch-target audit (Option 1) — closes the closest real usability gap.
  2. Status bar context labels (Option 2) — highest clarity payoff per pixel.
  3. Standardize cost/risk rubric (Option 5) — ties existing work together.
  4. Typography pass (Option 6) — quick win on readability.
  5. Event result summary (Option 3) — reduces post-event confusion.
  6. Accessibility contrast & focus-visible (Option 8) — finish the polish.

Decision Matrix

Fast wins

Options 1, 5, 6, 7. Deploy in any order; low risk, player-noticeable.

Structural clarity

Options 2, 3, 4. Require small UI additions; higher payoff on repeat play.

Quality floor

Option 8. One-time contrast/focus pass; fixes edge cases and inclusive play.

Verification Protocol

  • Every visual change must be browser-verified before being marked complete.
  • Use a real mobile device or emulated mobile viewport for touch-target checks.
  • If a change does not appear, treat it as a build/deploy issue — not a blind retry loop.
  • Document verdicts in CHANGELOG.md and HANDOFF.md after each verified pass.
