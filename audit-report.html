Métis Trail V1 — Production Audit & V2 Upgrade Brief

Rigorous technical review of the current build and a sourced roadmap to production-grade V2.

Executive Summary

The current v1 build is best understood as a vertical slice: solid thematic narrative, clear UI, and competent event writing. As an architecture, it is not production-ready. It ships as a single 1,600+ line HTML file with embedded state, data, and rendering logic. Scaling it will break fast. V2 already fixes the structural issues with modular ESM + esbuild; for highest impact, prioritize: (1) deterministic state/replay, (2) debugging tools, (3) scoring/LB, and (4) source-cited content pipeline.

1 — Architectural & Code Critique

1.1 What it is now

The v1 build is a single-file web app: 1,652 lines combining HTML, CSS, and two large embedded JS blocks (data + engine). It is playable, but it conflates every concern into one monolith.

1.2 Immediate red flags

  • Monolithic file structure — Data, engine, UI, and storage share one file. Any change risks regression across the entire stack.
  • No formal state layer — State is mutated ad-hoc inside closures. No snapshot/undo/replay; bugs are reproduced only by replaying input.
  • Randomness is not seeded for gameplay — Math.random() drives outcomes. This means identical choices yield different results, complicating QA and player trust.
  • Inline mutation patterns — Multiple functions mutate S directly. No immutability, no events, no logging of state transitions.
  • Single event pool — Events are static arrays. No weighting by season, node, crew state, or time pressure.
  • Persistence is absent — No save/load. The experience is single-session only.
  • Scoring is minimal — Final score is a simple linear formula. No meaningful differentiation between playstyles.
  • Offline/build maturity — V1 is not structured for offline bundling, dependency pinning, or reproducible local builds.

1.3 What V2 gets right

  • ESM modules with clear boundaries: core, data, systems, ui
  • Deterministic PRNG for reproducible runs
  • localStorage save/load scaffolding
  • Separation of calendar, items, events, nodes as data files writable by non-devs
  • Source citation objects attached to narrative content

1.4 V2 gaps that will still cause pain

  • State is still mutable global — Guild-style state object invites accidental mutation; add immutable reducers or a state machine before expanding.
  • Import graph not enforced — Data imports core cleanly, but ui imports systems with no contract/types (JSDoc is good, but not enforced).
  • No telemetry/trace layer — Still no event stream; debug overlay exists but is not wired to a logger.
  • CI/build discoverability — The Pages deploy issue shows lack of build artifact validation; you must verify dist/ exists before deploy steps consume it.

1.6 Anti-pattern to burn now

// v1 anti-pattern
function travelOneDay() {
  S.food--;           // implicit side effects on shared object
  S.day++;            // multiple FUNCTIONS can do this differently
  S.wear++;           // no record of WHO changed it
}

// v2 direction
function step(state, action) {
  const log = [];
  let next = structuredClone(state);
  next.food -= 1;
  if (next.food <= 0) { next.over = true; log.push({ kind: 'gameover', reason: 'food' }); }
  return { state: next, log };
}

2 — Robust Debugging & Bug Reporting Architecture

2.1 Core principle

Ship a first-class internal tool with a toggle. Developer UX is your multiplier. Bugs that cannot be reproduced in under 60 seconds never get fixed.

2.2 Debug suite blueprint

/**
 * @typedef {Object} DebugAction
 * @property {'warp'|'give'|'set'|'log'|'snapshot'|'replay'} kind
 * @property {*} payload
 */

class DebugConsole {
  constructor(store) {
    this.store = store;
    this.entries = [];        // append-only state log
    this.snapshots = [];      // sparse saved states
    this.bindUI();
  }
  dispatch(action) {
    const entry = { ts: Date.now(), action };
    switch (action.kind) {
      case 'warp': {
        const { day, node } = action.payload;
        this.store.state.day = day;
        this.store.state.node = node ?? this.store.state.node;
        entry.result = { day, node: this.store.state.node };
        break;
      }
      case 'give': {
        const item = this.store.cart.find(i => i.name === action.payload.name);
        if (item) item.count += action.payload.amount;
        entry.result = { item: item.name, count: item.count };
        break;
      }
      case 'set': {
        Object.assign(this.store.state, action.payload.fields);
        entry.result = action.payload.fields;
        break;
      }
      case 'snapshot': {
        this.snapshots.push({
          ts: entry.ts,
          state: structuredClone(this.store.state),
          cart: JSON.parse(JSON.stringify(this.store.cart))
        });
        entry.result = { count: this.snapshots.length };
        break;
      }
      default: entry.ignored = true;
    }
    this.entries.push(entry);
    this.render();
  }
  exportBugReport() {
    return JSON.stringify({
      meta: { ts: Date.now(), userAgent: navigator.userAgent },
      state: this.store.state,
      cart: this.store.cart,
      log: this.entries.slice(-50),
      lastSnapshot: this.snapshots.at(-1)
    }, null, 2);
  }
}

window.addEventListener('error', (e) => {
  const payload = {
    msg: e.message,
    file: e.filename,
    line: e.lineno,
    col: e.colno,
    state: store ? serialize(store) : null,
    stack: e.error?.stack
  };
  navigator.sendBeacon?.('/api/bug-report', JSON.stringify(payload));
  // Fallback: queue to localStorage and send on next clean load
  localStorage.setItem('metis_pending_bug', JSON.stringify(payload));
});

window.addEventListener('unhandledrejection', (e) => {
  console.error('Unhandled rejection:', e.reason);
  // same shape as above
});

2.4 User-facing bug report flow

  1. User presses hidden-dev-combo or taps Debug toggle
  2. System generates runtime JSON payload: state + last 50 log entries + last snapshot
  3. User copies from textarea or downloads metis-bug-YYYYMMDD-HHMMSS.json
  4. Dev recreates state with DebugConsole.restore(payload)

2.5 Speed-run / warp tooling

  • WARP node:5 day:120 seed:42 — jump to any node/day with fixed seed
  • LOG tail:20 — show last 20 state transitions
  • SNAPSHOT / RESTORE 3 — save/load sparse state
  • GIVE Pemmican 10 / SET wear:0 food:99 — inject resources
  • FORCE_EVENT broken_wheel — inject a specific event by ID

3 — Systems Design: Scoring & High Scores

3.1 Design goals

  • Multiple win conditions feel distinct, not just “same result with different numbers”
  • Punish inefficient but reward patient, historically plausible behavior
  • Surface historical literacy via scoring (e.g., visiting culturally significant sites)

3.2 Scoring formula (production-ready pseudocode)

/**
 * Base score decomposition:
 *   Base = 1000 (arrival at Edmonton)
 * Subtract:
 *   - 6 per day elapsed      # time pressure
 *   - 40 per wear point      # cart condition
 *   - 5 per camp made        # rest tradeoff
 *   - 50 if crew exhausted   # bad management
 * Add:
 *   + 120 per trade good
 *   + 75 per cultural milestone
 *   + 20 per food remaining
 *   + 30 per rested crew member
 * Multiply:
 *   x (1 + restBonus/1000)   # sublinear bonus for rest-heavy runs
 *

3.3 Implementation

function calcScore(state, cart) {
  if (!state.won) return 0;
  const tradeGoods = cart.filter(i => i.type === 'trade' && i.count > 0).reduce((s, i) => s + i.count, 0);
  const culturalVisited = [...new Set(state.nodesVisited.filter(n => ['metis','mission','nwmp'].includes(n.type)))].length;
  const restBonus = Math.min(state.camps * 8, 120);

  let score = 1000;
  score -= state.day * 6;
  score -= state.wear * 40;
  score -= state.camps * 5;
  if (state.crew === 'exhausted') score -= 50;
  score += tradeGoods * 120;
  score += culturalVisited * 75;
  score += state.food * 20;
  if (state.crew === 'rested') score += 30;
  score = Math.floor(score * (1 + restBonus / 1000));
  return Math.max(score, 0);
}

3.4 Persistent leaderboard options

A. Pure client-side (fastest)

Store top-N in localStorage. Zero network deps. Pro: works offline, no infra. Con: per-browser, can be cleared.

B. Serverless JSON file

POST to GitHub Gist or a Cloudflare Pages function writing a JSON file. Pro: real persistence. Con: auth + rate limits.

C. GitHub Issues as scores

Create issue with label score via GitHub API. Pro: auth-free public, auditable. Con: workspace, noise.

D. Firebase/Supabase

Lightweight row in a hosted table. Pro: proper global LB, auth. Con: backend knowledge, billing.

Recommendation: Start with A, introduce B or C only if a global community emerges. For V2, make the leaderboard interface-agnostic: an adapter with one method saveScore(entry) and one getTop(n). You can swap implementations without touching UI.

4 — Evolving from Novelty to a Full Game

4.1 Current loops (v1)

  • Travel loop: click Travel/Make Camp → narrative text + random event overlay → choice → effect
  • Settlement loop: reach node → action buttons → trade/rest/repair → continue
  • Cart loop: wear increases from events/travel; repair costs shaganappi

These loops are functional but shallow. Replayability is low because events do not meaningfully interact with each other.

4.2 Depth patterns to add

  • Seasons as difficulty ramps: map has a daily “stress” value that rises after August. Food depletion accelerates; rivers swell (more crossing events).
  • Inventory weight as first-class mechanic: heavy cart = slower speed = longer exposure to random events. Deliberate curation is rewarded.
  • Moral dilemmas with delayed consequence: e.g., an ailing traveller offers a map for half your food. Save them and gain a route shortcut later; ignore them and find no help at the next river crossing.
  • Encounter chains: events that reference prior choices (Deserter → recognised by NWMP? Same river crossing remembered by ferry operator?).
  • Economic simulation: Commodity prices shift by node and season. Pemmican cheap at Batoche, expensive at Edmonton.
  • Permadeath + legacy: on loss, unlock a short “Trail Tales” note about what happened to the crew.

4.3 Pacing rhythm

Phase structure per run:
  1. Setup (intro)           — 90 seconds
  2. Early Prairie           — 5 minutes, events are resource-positive
  3. River Valley mid-game   — 10 minutes, events become tradeoffs
  4. Wooded/Upland late game — 10 minutes, risks escalate, fewer safe choices
  5. Endgame sprint          — 5 minutes, Edmonton reachable, winter pressure

4.4 Replay design

  • Rotating event pools by seed so no two runs feel identical
  • Daily narrative log with “highlights reel” for social sharing
  • Scoring tiers that reward different archetypes: “Steady Brigade,” “Daredevil,” “Trader,” “Historian”

5 — Deep Historical Accuracy & Source Material Respect

5.1 What V1 gets right

  • Landscape framing — buffalo herds, prairie fire, river fording, mosquito plagues
  • Culturally specific details — shaganappi, bannock, saskatoon preserve, Sunday song references (“Old Hundred, Balerma, Dundee, Ward”)
  • NWMP tension and duty narrative
  • Red River cart squeal as a persistent mechanic

5.2 Where it flattens the history

  • Crew is singular, not collective — “Your crew” sounds like one entity. Real freight brigades were family groups and Métis kinship networks.
  • No Michif language or bilingual texture — Even a handful of Métis terms (e.g., “s_voyer_,” place names) would ground the world deeper.
  • Events read like generic survival fiction — Several are modular D&D-style encounters; they reference 1878 Western Canadian history only loosely.
  • Missing economic context — Why pemmican? Because the HBC supply pipeline was collapsing and Métis brigades filled it. Why is the Factor paying for beaver? Because fashion demand shifted; no mention of this in-game.
  • Settlements are atmospheric but not structural — Fort Garry, Batoche, and Edmonton are described beautifully, but their historical roles in 1878 are not reflected in gameplay differences.

5.3 Integrating history into mechanics

  • Religion as resource: Sunday rest at St. Norbert/Batoche restores crew differently than at an HBC post (missions provide morale/health; forts provide repair/trade only).
  • Family obligations: Some events require choices with kinship cost. Example: your sister begs you to take her child west for safety; that consumes a food/weight slot but unlocks crew loyalty at a future node.
  • Trade good historicity: Replace “trade goods” with historically grounded outputs: pemmican, dry meat, bison robes, beaver pelts, wolf skins. Their Edmonton market value changes with supply.
  • River crossing seasonality: June-July crossings are risky (melt). August-September safer. Mechanically, change fording DC by month.
  • Treaty/papers: Introduces travel documentation as an inventory item. At NWMP posts, you must present papers. Losing them creates a cascading penalty.
  • Citation tooltips: Every narrative passage should expose its source on hover or via a “Read source” link. V2 already has the source registry; wire it into the UI.

5.4 Specific historical recommendations

  • Introduce Cuthbert Grant / Gabriel Dumont figures as non-combat NPCs with region-specific requests (e.g., Gabriel Dumont at the South Saskatchewan offers a shortcut for carrying dispatches).
  • Reference the 1878 bison collapse era: fewer herds, more hunting pressure — make game_sighted events rarer as days increase.
  • Use HBC post factorledger logs as a known pricing curve (edmonton prices go up later season; batoche prices drop as locals glut supply).
  • St. Norbert, St. Boniface, St. François Xavier are actual parish nodes; use their historically documented functions (school, grist mill, ferry).
  • Place calendar Easter, St. John’s Day, and winter count references in travel flavor.

6 — Prioritized Action Plan

Phase 1 — Foundation & Technical Debt (Weeks 1–3)

  • Finish V2 modular build and Pages deploy. Build artifact validation; rollback plan.
  • State machine + reducer in src/core/state.js. Replace S.mutate() style with step(state, input).
  • Pure engine in src/systems/engine.js: no DOM access. All interactions via a clean API.
  • Logger: append-only event stream tied to each turn.
  • Save/load with schema validation.
  • Tests for calendar, scorer, PRNG, event resolution.

Phase 2 — Core Systems & Debugging (Weeks 4–7)

  • Weight system: total weight affects speed/wear; show live weight bar.
  • Component-based cart wear: wheel/axle/tongue subsystems squealing thresholds.
  • Crew morale + hunger with per-member modeling.
  • Debug suite implementation from Section 2 with warp/give/snapshot/replay.
  • Scoring formula V2 with tiered mastery labels.
  • Bug report plumbing (localStorage queue + exportBugReport()).
  • Offline-ready dist/ with bundled assets; remove CDN dependency.

Phase 3 — Content & Mechanics Expansion (Weeks 8–16)

  • Seasonal event weighting and node-specific encounters.
  • Economic simulation: dynamic pricing, supply g.
  • Historical narrative layer: source-cited events, day journal.
  • Leaderboard adapter: choose A/B/C based on user feedback.
  • Accessibility pass: keyboard nav, reduced-motion mode, text-only fallbacks, ARIA labels.
  • Polish: icons for inventory, map illustrations, audio cues (optional), journal export as PDF.

Definition of Done per phase

  • Phase 1: npm test green, npm run build always produces dist/, GitHub Pages deploy verified.
  • Phase 2: can speed-run any 10-minute test scenario using debug tools, scoring formula described above implemented, cart wear and crew hunger feel meaningfully different.
  • Phase 3: 3+ distinct viable strategies, 30+ source-cited events, first external playtest with documented feedback.

Final Verdict

V1 is a strong vertical slice. The writing, theme, and UI shell are genuinely good. Treat it as a prototype, not a product. V2 is the correct evacuation path. Execute in the order above: stabilize architecture, instrument the system, then enrich content. Do not add features to V1.
